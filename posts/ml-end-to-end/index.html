Machine Learning End-to-end

Data Science is a field of study that has been developing at a fast pace in the last years as a result of considerable research in the Machine Learning area, but also by the impact that it has been causing in our society. With this in mind, companies are investing in AI creating research teams to innovate their products. Nevertheless, the produced models must be productized to deliver real-time predictions and create business impact. With this intention, Software Engineers with Machine Learning skills are needed to architect and build Machine Learning pipelines that allow real-time predictions. These individuals that usually have Computer Science background allied with AI, Maths and Statistics are the Machine Learning Engineers. Data Scientists (that ideally also have Engineering skills) must work with ML Engineers to define the pipeline and productize their research. In this blog post, I’m sharing some content related to the role of and ML engineer in a company (see [4] and [5]) and I’m detailing on how to productize Machine Learning (because this is the skill that the Data Scientists need to acquire üòâ).

Machine Learning Lifecycle

Figure 1 shows an example of an ML pipeline. While most Data Scientists are familiarized with the steps one to six, the last two may be challenging for the most part. However, while for the most times Researchers may be working offline (i.e., on their notebooks with previously collected data), in order to productize their models, Machine Learning engineers must architect this pipeline to feed the models with real-time data. This leads us to the definition of two important components:

  • Online Model Analytics: the model is applied for real-time decision making;
  • Offline Data Discovery: analysis on historical data to create the ML model in a batch-processing mode.

Figure 1: Machine Learning Pipeline illustrated by Semi Koen in [1].

Figure 2 shows a more detailed pipeline that includes both the online and offline layers. Detailed information regarding each pipeline component may be found in [1].

Figure 2: Machine Learning End-To-End illustrated by Semi Koen in [1].

Observing the figure above, we observe that the Model Scoring (also known as Model Serving) is the most complex stage for the Machine Learning Engineers. This phase implies that the model was previously deployed and embedded in a decision-making framework. The deployment is challenging as, traditionally, the programming languages needed to operationalise models are different from those used to develop them. One solution is to deploy the model to a container and run as a Microservice. This operation often requires continuous delivery implementation, forcing the new MLOps role [3].

Finally, according to Semi Koen, the main objectives of building Machine Learning systems are:

  • Reduce latency;
  • Integrated but loosely coupled with the other parts of the system, e.g. data stores, reporting, graphical user interface;
  • Can scale both horizontally and vertically;
  • Are message driven i.e. the system communicates via asynchronous, non-blocking message passing;
  • Provide efficient computation with regards to workload management;
  • Are fault-tolerant and self healing i.e. breakdown management;
  • Support batch and real-time processing.

Abstract the Pipeline

Fortunately, nowadays, there are several platforms to facilitate the Machine Learning Productization, helping Data Scientist to create these pipelines fast and efficiently. Some examples of these platforms are MLflow, Metaflow and Azure ML Studio (check Extra Resources for links).

Ultimately, an example of an end-to-end Machine Learning project is described in [2]. In this case, they use MLflow to keep track of the model version and AWS Sagemaker to deploy the model.
